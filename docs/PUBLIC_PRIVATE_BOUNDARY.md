# Public / Private Boundary

Faith Team Ops follows a simple rule:

Public surface. Private engine. Receipts always.

## Public

The public repository may include:

- Project purpose and audience.
- High-level workflow diagrams.
- Visual concept boards.
- Public status.
- WordPress page copy.
- Ownership, security, and commercial-use policies.
- Privacy review and launch checklist.

## Private

The public repository must not include:

- WordPress plugin source.
- Release zip packages.
- Credentials, secrets, tokens, keys, cookies, or environment files.
- Deployment systems, server configuration, backups, or database dumps.
- Internal prompts, agent instructions, private automation workflows, or source-of-truth logs.
- Private team records, support requests, private media, voice notes, audit data, legal/admin files, medical files, benefits records, family information, or unpublished strategy.

## Review Principle

If a file proves the project without exposing the engine, it can be considered for public use.

If a file would let someone reconstruct private operations, impersonate access, copy strategy without attribution, or infer sensitive records, it stays private.
